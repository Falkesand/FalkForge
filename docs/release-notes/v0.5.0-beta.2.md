# FalkForge v0.5.0-beta.2

This is the second public beta. It carries no new authoring surface — the focus was closing a
class of bug found across an honesty sweep of the compiler, engine, and CLI: inputs that were
accepted and validated, then silently dropped instead of doing what they said, or doing
something wrong without telling you. Every fix below either makes the accepted input actually
work, or turns the silent drop into a loud, discoverable error/warning so a green build never
hides missing behavior.

## Fixed: silently dropped input now does what it says

- **ARP metadata** — `Comments`/`Contact`/`HelpUrl`/`AboutUrl`/`UpdateUrl` on the package model
  were accepted but never written to the MSI `Property` table; Add/Remove Programs showed none
  of them. Now emitted correctly.
- **Bundle UI branding** — logo, theme, watermark, and banner set on the bundle manifest were
  parsed but never reached the installer window. Now wired through end to end.
- **Download throttle** — `DownloadThrottle` was accepted but never wired into
  `PayloadDownloader`; throttled downloads ran at full speed. Now actually throttles.
- **Reproducible-build SBOM epoch** — a `Reproducible()` epoch override was honored in build
  timestamps but not carried into the SBOM's own identity/serial derivation, for both MSI and
  bundle outputs, so two builds with the same override could still emit SBOMs with different
  identities. Both paths now derive the SBOM identity from the same epoch.
- **License-file-without-license-page** — setting `LicenseFile` on a dialog set with no license
  page silently dropped the license text; the file was compiled in but never shown. Now warns.
- **MSI license text rendering** — the license text from `LicenseFile` was silently dropped for
  MSI output (the RTF control never got the content). Now renders correctly.
- **Authenticode signature across bundle detach/reattach** — detaching and reattaching a bundle's
  integrity manifest (sign-then-reattach workflows) stripped the outer Authenticode signature by
  truncating the PE security directory. The self-extraction still worked, so this was invisible
  until someone checked signtool. Fixed by extending the security directory instead of truncating
  it.
- **Download throttle below the read-buffer size** — `TokenBucket` capped burst capacity at
  `bytesPerSecond`, so any `DownloadThrottle` rate set below `PayloadDownloader`'s 81920-byte read
  buffer meant a single full-chunk token request could never be granted; the download would idle
  out and fail after three retries instead of throttling. `TokenBucket` now takes an independent
  burst-capacity floor sized to the read buffer, so low throttle rates work instead of failing.
  (Same commit hardens `ManifestBranding` theme-color parsing to catch the full set of exceptions
  `ColorConverter` can throw, so a bad author-supplied color is ignored instead of crashing UI
  startup.)

## Fail-loud: accepted-but-not-applied inputs now error or warn instead of vanishing

Six inputs were validated (or carried) and then silently ignored. Each now surfaces a
discoverable diagnostic through its layer's existing mechanism:

- **JSN019 (CLI JSON loader)** — a JSON `extensions` block (Firewall/IIS/SQL/.NET) is
  **not supported** by the declarative JSON authoring path. Previously a well-formed block
  validated cleanly and was then dropped, producing an installer silently missing the firewall
  rules, IIS site, SQL database, or .NET check the author declared. `forge build`/`forge validate`
  now **fail the build** with JSN019 once the block carries any content — field validation
  (JSN011–JSN014) still runs first. Use the C# fluent API for these four extensions
  (`new MsiCompiler().Use(extension)`); see the demo catalog's "Extensions Showcase" and the
  focused Firewall/IIS/SQL/.NET demos.
- **BDL034 (bundle compiler)** — per-package `EnableFeatureSelection` is carried but never
  becomes MSI `ADDLOCAL`; now warns that the package installs its default feature set.
- **BDL035 (bundle compiler)** — a container `DownloadUrl` is stored but every payload is
  embedded in the single exe; now warns that external containers are not produced.
- **EXT002 (MSI authoring)** — the `IComponentContributor` drop warning used to be silent for
  null-logger callers; now always emitted (structured logger when present, stderr fallback
  otherwise).
- **EXT003 (MSI authoring)** — registry-registered `IDryRunContributor`s were discarded by an
  empty no-op; now collected and warned (dry-run previews come from `forge build --dry-run` /
  `forge validate`, not this pipeline).
- **DLG005 (dialog producer)** — more than one localization culture implies per-culture MST
  output that FalkForge does not generate; now warns that only the first culture is applied.

If you were relying on any of the six behaviors above silently no-op'ing, your build now fails
or warns where it previously succeeded quietly — that is the intended, honest outcome.

## Known Limitations

Carried forward from beta.1, still true:

- **MSIX is experimental.** `Compiler.Msix` is wired into the fluent API and Studio, but there's
  no CLI dispatch yet (`forge build` doesn't produce `.msix`/`.msixbundle`). Use it for feedback,
  not production packaging.
- **IIS certificate bindings and sub-applications are declared but not yet enforced.** They're
  written into site configuration; the validator warns (IIS013/IIS014 for bindings, IIS015 for
  sub-apps) but nothing binds the certificate or creates the sub-application at install time.
- **`Reproducible()` + `Integrity()` on bundles isn't byte-identical yet.** On MSI, combining the
  two automatically moves the signature to a sidecar file so the MSI itself stays reproducible.
  Bundles don't have that guard: `BundleBuilder.Integrity()` always signs in-band, so a
  reproducible+signed bundle build is not byte-identical across rebuilds today. Workaround: build
  unsigned with `FALKFORGE_NO_SIGN=1` and sign out-of-band.
- **`forge verify` for MSI is classical-only.** `--trusted-key` pins an ECDSA fingerprint; there's
  no `--pq-trusted-key` equivalent for the hybrid ML-DSA companion signature yet (hybrid pinning
  at the engine-trust level, via `EngineTrustAnchor.TrustHybridKey`, does exist — this gap is
  specifically in the `forge verify` CLI path).
- **Signature expiry windows and a transparency log are designed but not shipped.** No automatic
  aging-out of a retired key's signatures, and no public tamper-evident log of issued signatures
  yet.
- **JSON configuration cannot author extensions.** Firewall, IIS, SQL, and .NET-detection
  configuration is only available through the C# fluent API — see the fail-loud JSN019 entry
  above. This is a permanent scope boundary for the JSON path, not a temporary gap.

## Requirements

- Windows, to *build* MSI/MSM/MSP/MST output (P/Invokes `msi.dll`). Bundle compilation and the
  decompiler's bundle path are cross-platform; MSI compilation, inspection, and the WiX-Burn
  decompiler are Windows-only.
- .NET 10 SDK.
