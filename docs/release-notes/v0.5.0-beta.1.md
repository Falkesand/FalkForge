# FalkForge v0.5.0-beta.1

This is the first public beta of FalkForge: a C# fluent-API framework for building Windows installers. You author an installer as a strongly-typed C# object graph — packages, features, files, registry, services, custom actions — and compile it to a real MSI (or MSM/MSP/MST) via a P/Invoke wrapper around `msi.dll`, or to a self-extracting EXE bundle driven by a NativeAOT engine with its own WPF UI. No WiX XML in the loop.

The repo is going public with this tag. It's beta software: the core compiler and engine are exercised by ~3500 tests and used on real installs, but APIs can still shift and some features (flagged below) are intentionally incomplete. If you hit a bug, a confusing API, or a gap between what the manual promises and what actually happens, please file it — that mismatch is exactly what this beta is for. Feedback on the fluent API's ergonomics is just as welcome as bug reports.

## Getting Started

Pick one:

```bash
# Scaffold a standalone project from a template
dotnet new install FalkForge.Templates
dotnet new falkforge-msi -n MyInstaller --ProductName "My App"
# or: dotnet new falkforge-bundle -n MyInstaller --ProductName "My App"

# Or add the meta-package to an existing project
dotnet add package FalkForge

# Or scaffold in-place from the CLI
forge init --name "My App"
```

- Docs: https://falkesand.github.io/FalkForge/ (also attached to this release as a docs zip)
- Bugs and questions: GitHub Issues on this repo

## Highlights

- **Fluent, type-safe authoring** — `PackageBuilder`, `FeatureBuilder`, `FileSetBuilder`, etc. build a `PackageModel` in C#. `MsiProperty` and `Condition` are typed wrappers (not raw strings) for MSI property refs and conditions, with operators (`&`, `|`, `!`) and pre-composed conditions like `IsPrivileged`, `Is64BitOS`, `IsInstalled`.
- **Full output range** — MSI, MSM (merge modules), MSP (patches), MST (transforms), and self-extracting EXE bundles, all from the same model.
- **3-process NativeAOT engine** — UI, Engine, and an elevated companion process talk over named pipes (HMAC-authenticated handshake). Phases: Detect → Plan → Elevate → Apply → Complete, with a rollback journal and undo operations if anything fails mid-apply.
- **Extensions** — Firewall, IIS (app pools, web sites, bindings, certificates), SQL Server (DB creation, script execution), .NET runtime detection, HTTP.sys URL reservations + SNI SSL bindings, device drivers (PnP/INF), plus Util and Dependency (WiX-compatible provider/consumer ref-counting). These aren't inert MSI rows — Firewall, SQL, IIS, HTTP, Dependency, and Util schedule real deferred, elevated custom actions that do the work at install time.
- **Custom UI** — build a fully custom WPF installer UI (`InstallerPage<TView>` + `InstallerApp.Run()`), or author custom native MSI dialogs directly (`PackageBuilder.AddCustomDialog(...)`) with controls, events, and conditions, compiled straight into the MSI dialog tables.
- **Integrity signing** — pure .NET ECDSA-P256 payload signing (`Integrity()`) for both MSI and bundles, no external tooling required. Optional hybrid post-quantum companion signatures (ML-DSA-65) alongside the classical key. `forge verify` checks signatures against pinned trusted-key fingerprints, with content binding so a payload swap or table edit is caught.
- **Reproducible builds + SBOM** — `Reproducible()` pins builds to `SOURCE_DATE_EPOCH` for byte-identical output; a CycloneDX SBOM sidecar with per-file SHA-256 is generated alongside.
- **Delta updates** — `BundleBuilder.DeltaFrom(oldBundle)` generates Octodiff-based delta bundles; the engine downloads delta-first and falls back to a full bundle automatically.
- **WinGet manifests** — `PackageBuilder.WinGet(...)` generates a 3-file WinGet manifest (version/installer/locale YAML) alongside the MSI, with the SHA-256 computed at compile time. `forge winget` retrofits a manifest onto an existing MSI.
- **Decompile / migrate from WiX** — `forge decompile` turns an existing MSI or WiX-Burn bundle back into a buildable C# project; `forge migrate` wraps that into a one-shot MSI/WiX → FalkForge migration workflow.
- **Localization** — JSON-based, with a culture fallback chain (specific → parent → default). `forge loc export` dumps the built-in dialog strings as a starting point for your own translations/overrides.
- **65 worked demos** in `demo/` covering everything above end to end, from a minimal single-file MSI to hybrid PQ signing and delta updates.

## Known Limitations

- **MSIX is experimental.** `Compiler.Msix` is wired into the fluent API and Studio, but there's no CLI dispatch yet (`forge build` doesn't produce `.msix`/`.msixbundle`). Use it for feedback, not production packaging.
- **IIS certificate bindings and sub-applications are declared but not yet enforced.** They're written into site configuration; the validator warns (IIS013/IIS014 for bindings, IIS015 for sub-apps) but nothing binds the certificate or creates the sub-application at install time.
- **`Reproducible()` + `Integrity()` on bundles isn't byte-identical yet.** On MSI, combining the two automatically moves the signature to a sidecar file so the MSI itself stays reproducible. Bundles don't have that guard: `BundleBuilder.Integrity()` always signs in-band, so a reproducible+signed bundle build is not byte-identical across rebuilds today. Workaround: build unsigned with `FALKFORGE_NO_SIGN=1` and sign out-of-band.
- **`forge verify` for MSI is classical-only.** `--trusted-key` pins an ECDSA fingerprint; there's no `--pq-trusted-key` equivalent for the hybrid ML-DSA companion signature yet (hybrid pinning at the engine-trust level, via `EngineTrustAnchor.TrustHybridKey`, does exist — this gap is specifically in the `forge verify` CLI path).
- **Signature expiry windows and a transparency log are designed but not shipped.** No automatic aging-out of a retired key's signatures, and no public tamper-evident log of issued signatures yet.

## Requirements

- Windows, to *build* MSI/MSM/MSP/MST output (P/Invokes `msi.dll`). Bundle compilation and the decompiler's bundle path are cross-platform; MSI compilation, inspection, and the WiX-Burn decompiler are Windows-only.
- .NET 10 SDK.
