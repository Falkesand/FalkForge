# FalkForge v0.5.0-beta.3

This is the third public beta. Where beta.2 closed a class of silent-drop bugs, beta.3 turns
several of the accepted-but-not-yet-real inputs from beta.1/beta.2 into working features: the
JSON authoring path now actually attaches the firewall/IIS/SQL extensions it validates, the IIS
extension applies its web-application and certificate-binding configuration at install time, the
turnkey built-in UI host launches for real, and the interactive per-package MSI feature picker
lands end to end on the UI side. A handful of honesty fixes round out the release, in the same
spirit as beta.2: an input that can't do what it claims now fails loud instead of quietly doing
the wrong thing.

## Shipped this beta

- **JSON configuration can now author firewall/IIS/SQL extensions.** Previously any
  `extensions` block containing firewall/IIS/SQL/.NET content failed the build with JSN019 the
  moment it carried content. `JsonConfigLoader` now translates a present `firewall`/`iis`/`sql`
  block into the same real extension the C# fluent API attaches via
  `new MsiCompiler().Use(extension)`, reusing the extensions' own builders rather than
  re-implementing MSI emission, so the JSON path and the C# path produce identical tables.
  `.NET` runtime detection remains a hard error (JSN019): it's a bundle-engine, detect-time-only
  feature with no standalone-MSI representation, and the error message points authors at the C#
  fluent API (`new DotNetExtension().SearchForRuntime()...`) instead. Demos 06
  (`06-web-server.json`) and 07 (`07-database-app.json`) are restored with working
  firewall/IIS/SQL blocks and verified to build; docs (`documentation.html`, the JSON-config
  tutorial, the demo README) were corrected to match.
- **IIS extension applies WebApplication creation and HTTPS certificate binding at install
  time.** These were previously authored into the compiled MSI's configuration metadata but
  never applied — IIS013/IIS014 warned that they were deferred. Application pools, sites (with
  all bindings), and virtual directories were already live via deferred, elevated
  Microsoft.Web.Administration custom actions; sub-applications and certificate bindings now go
  through the same mechanism. Each authored WebApplication becomes its own deferred, elevated
  create step, sequenced after its site and before that site's virtual directories, with
  tolerant rollback/uninstall removal. Each HTTPS binding that references a certificate gets a
  dedicated binding step that locates a pre-provisioned certificate via the `Cert:` provider and
  applies its hash and store name — FalkForge binds a pre-provisioned certificate, it does not
  import one, and fails loud if the certificate is absent from the target store. Validation
  moved with the behavior: IIS013 is repurposed to the honest pre-provisioning caveat, IIS014
  now enforces `WebApplication.Alias`, the new IIS018 enforces `WebApplication.Directory`
  (parity with IIS016/IIS017 for virtual directories), and IIS015 now fires only when a virtual
  directory's non-root parent isn't an authored WebApplication.
- **Turnkey built-in UI host launches for real.** `FalkForge.Ui.exe`, spawned by the bundle
  engine with `--manifest`/`--pipe`/`--secret-pipe`, now parses those arguments, loads the
  manifest, connects an `EngineClient` back to the spawning engine over the data pipe, and shows
  the built-in wizard window. A missing or unreadable manifest fails loud with a message box
  instead of showing a blank window. This is the engine-first counterpart to the existing
  UI-first path used by custom-UI installers (demos 11-14), which spawns its own engine instead
  of being spawned.
- **Interactive per-package MSI feature picker — protocol, engine, and UI wired and unit-tested.**
  The bundle UI now renders a tree of selectable features for each MSI package the engine
  advertises, and sends the checked-feature-id set back so that package's `ADDLOCAL` gets
  stamped. This landed in layers: new protocol messages for advertising a package's feature
  table and sending back a selection; an engine-side reader for a wrapped MSI's feature table;
  engine-side application of the selection as `ADDLOCAL`; and the UI tree (parent/child
  sections, hidden/absent features excluded and their visible children re-parented to root),
  inserted into the wizard only for a package that actually advertised features.
- **Extension component-contributed files now emit into the compiled MSI**, and **extension
  dry-run actions now surface in `forge build --dry-run`** — both were previously collected
  internally and then discarded by a no-op.
- **Decompiler round-trips built-in UI branding fields** (logo, theme, watermark, banner) that
  beta.2 wired from the bundle manifest into the installer window — the decompiler previously
  dropped them when reconstructing a bundle from a built exe.
- **`forge build --format msix` fails loud instead of silently falling back to an MSI.**
  Previously it printed an "experimental, not yet implemented" note and built an MSI anyway
  under a `--format msix` request. It now returns a validation failure and points authors at
  calling `InstallerMsix.BuildMsix()` / `InstallerMsix.BuildMsixBundle()` directly from a
  build script.
- **Housekeeping:** removed the dead `coverlet.collector` package references across 45 test
  projects (a no-op under xunit.v3's Microsoft Testing Platform; coverage runs via
  `dotnet-coverage`, see the C6 coverage baseline work); the CLI's build logger is now disposed;
  and diagnostic fallback branches in the CLI build path picked up direct test coverage.

## Known limitations / in progress

- **The feature picker is dormant at runtime.** The UI side (protocol messages, tree builder,
  wizard page, `ADDLOCAL` application) is fully wired and unit-tested against advertised data,
  but the engine-side detect-time advertising isn't wired yet: the bundle extraction root isn't
  plumbed to the pipeline's detect step, and extracted-MSI path resolution is part of the
  still-stubbed live bundle-install path. No `PackageMsiFeaturesMessage` is emitted at runtime
  until that lands. Static per-package feature selection works today by setting the standard MSI
  `ADDLOCAL` property directly, e.g. `.Property("ADDLOCAL", "F1,F2")`.
- **The turnkey UI host and the new IIS install-time actions are wired but only exercised live
  under `FALKFORGE_REAL_SYSTEM_E2E`** on a real machine with an elevated shell (IIS also needs a
  real certificate in the target store) — neither path runs in CI.
- **Downloadable external containers (`Container.DownloadUrl`) remain unimplemented.** The URL
  is stored but every payload is still embedded in the single bundle exe; BDL035 warns that
  external containers aren't produced. Carried forward from beta.1/beta.2, unchanged this
  release.

## Requirements

- Windows, to *build* MSI/MSM/MSP/MST output (P/Invokes `msi.dll`). Bundle compilation and the
  decompiler's bundle path are cross-platform; MSI compilation, inspection, and the WiX-Burn
  decompiler are Windows-only.
- .NET 10 SDK.
