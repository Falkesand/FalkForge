Bottom line

FalkForge is genuinely pleasant to start — the fluent C# API hides every MSI internal (GUIDs, Components, KeyPath, the Directory table) that makes WiX painful, and a simple single-tool MSI is a ~15–20 minute job. But this audit surfaced one real code bug that is bigger than any doc gap: the extension table-contributor pipeline is unwired, so IIS / SQL / Firewall / etc. configuration never reaches the compiled MSI — live-proven, not just code review. The enterprise story is authorable but doesn't actually compile through today. Separately, the manual is a ~95% reference wall with no "concepts" teaching and the best tutorial content is orphaned. Core MSI authoring is solid and proven working — the problems are scoped, not systemic.

1 · Journey A — a simple installer for a small Windows tool

Persona goal: ship an .msi for a single PomodoroTimer.exe — Program Files, a Start Menu shortcut, clean uninstall.

Verdict

Once you can actually get the tool, this is the framework at its best. Scaffold → dotnet run is ~2 min; the only real work is adding one shortcut call. No MSI concepts leak out — no GUID, no Component, no KeyPath. Uninstall is free.

2 · Journey B — the full-blown enterprise installer

Persona goal: "AcmeSuite" — a feature tree (Server / Client / Tools), a Windows service, registry config, an IIS site + app pool, a SQL Server database, a custom action, code signing, and an EXE bundle wrapping prerequisites + the MSI with an update feed.

Verdict

Cannot be assembled from docs + demos today, for two independent reasons: (1) the extension capabilities the enterprise case depends on (IIS, SQL) silently produce nothing — see §3; and (2) even where capabilities work in isolation, there is no end-to-end example that composes them, and feature-gating a service/registry entry isn't reachable through the fluent API.

Two structural gaps beyond the bug

  • gap No "put it all together" enterprise example. Checked all 60+ demos, docs/tutorials/, the manual, and the MAS reference app — the richest single artifacts (doc §7.27 "Complete Example"; demo 05 enterprise-suite; demo 06 product-suite) each cover a subset. Nothing combines feature-tree + service + registry + IIS + SQL + custom-action + bundle + signing.
  • gap · confirmed Can't feature-gate a service/registry entry via the fluent API. ServiceModel.FeatureRef / RegistryEntryModel.FeatureRef exist but have no builder setter, and there is no FeatureBuilder.Service(...)/.Registry(...). 3 · The extension bug — live-proven, the headline finding

Claim, escalated to proof

live-proven Extensions that emit MSI tables (IIS, SQL, and by the same code path Firewall / Dependency / Driver / Http / Util) configure successfully, print success, and produce an MSI missing their tables entirely. This is a false-positive success: a newcomer following demo 30/31 ships a broken installer believing it works.

The live evidence

The control proves the inspector reflects real config — so "ABSENT" for SQL/IIS is a genuine absence, not a tooling artifact.

Root cause (unwired pipeline, self-documented)

// src/FalkForge.Compiler.Msi/Recipe/MsiAuthoring.cs:174-178
// "No other contributors wired in yet — phase 11 will route
//  IMsiTableContributor through here."
contributors: Array.Empty<IMsiTableContributor>()

// same file :492 — the registry the extensions call into is a no-op
public void RegisterTableContributor(IMsiTableContributor c) { }   // drops it
  • All 6 extension demos call a 0- or 1-arg new MsiCompiler(...), so the compiler's extension list is empty; but even passing extensions correctly wouldn't help, because the contributor array is hardcoded empty downstream. The gap is framework-wide, not per-demo.
  • SqlExtension.Register() does call RegisterTableContributor(...) correctly — the contribution lands in the no-op and is dropped silently (src/FalkForge.Extensions.Sql/SqlExtension.cs:18-23).
  • IisExtension.Register() is a literal no-op and IIS declares no contributor type at all — deeper gap than SQL (src/FalkForge.Extensions.Iis/IisExtension.cs:23-28).
  • Zero warning surface: BuildCore validation only inspects the PackageModel; the standalone extension objects are never seen, so orphaned IIS/SQL config produces no error or warning.
  • Why tests didn't catch it: extension tests cover builders/models/validators in isolation; nothing asserts end-to-end that the table appears in a compiled MSI.

4 · Documentation review — meaty, but the wrong kind of meaty

README good front door

217 lines, tight, a 30-second "Three Ways to Build" table, forge init featured first, a working Quick Start snippet, a genuine Security section, links out to demos + manual. No complaints — this is the strongest single onboarding asset.

documentation.html reference wall

  • 11,271 lines, 22 sections. ~95% API reference, ~0% tutorial. Section 7 (MSI Builder API) alone is ~1,812 lines of one-subsection-per-builder signature tables. 244 API tables across the file.
  • No "Concepts" section. Searched for concept/glossary/component rule/why guid/mental model — zero hits. GUID/Component/ProductCode appear only inside parameter tables. The manual assumes the Windows Installer mental model and teaches none of it. A WiX veteran reads fine; a true beginner gets syntax without meaning.
  • forge init: zero mentions in 11k lines (README/doc divergence, repeated from §1).
  • Navigation tooling itself is good — sidebar, chevrons, search box, dark/light, deep anchors. The problem is content shape, not findability.

Demos as a learning ladder strength / one nit

63 demos, but not a flat dump: demo/README.md (831 lines) groups them into tables with feature matrices, 61/63 have their own README, numbering runs beginner→advanced. This is genuinely better than most sample repos. Nit: no "if you're new, read 01 → 02 → 06 → 11 and stop" callout above the 150-row matrices.

Orphaned gold fix

docs/tutorials/ (14 files, ~4,084 lines) holds the best narrative content in the repo — getting-started, coming-from-wix, a services deep-dive with real LocalSystem-vs-NetworkService security tradeoffs. The README never links it, so the one place that actually teaches is invisible on the documented path.

5 · Good parts — honest credit

  • The fluent API is the product's best asset. Plain typed C#, IntelliSense, no XML, no MSI internals leaking. Proven to hide GUID/Component/KeyPath entirely on the simple path.
  • Security/signing depth is real and rare for an installer tool — payload-hash signing, hybrid ML-DSA-65 PQ, trust roles/quorum, revocation, anti-downgrade, reproducible builds/SBOM. §22 of the manual is a model tutorial (WHY, with diagrams).
  • Demo catalog is well-indexed — grouped tables, feature matrices, per-demo READMEs, consistent numbering.
  • Thoughtful CLI ergonomics — forge init clobber-protection (check-all-before-write, --force), --from-publish prefill.
  • Honesty in the docs where it counts — the update-feed section admits unimplemented policy behavior rather than overselling.
  • Design care where wired — typed cross-refs (AppPoolRef, CertificateRef, SqlDatabaseRef, ContainerRef), 8 extensions, forge migrate for WiX refugees.

6 · Prioritized fixes

P0 — correctness (do before the enterprise story is credible)

Wire the IMsiTableContributor pipeline ("phase 11"). Route registered contributors into MsiAuthoring so SQL/Firewall/Dependency/Driver/Http/Util tables actually emit; give IIS a real contributor (or custom-action emitter). This is the single highest-impact fix — it turns 6 "working-looking" demos from broken to real. MsiAuthoring.cs:178, :492
Add a fail-loud guard for orphaned extension config. If an extension is configured but nothing it contributes reaches the model, warn/error at build. Silent false-success is worse than a missing feature.
Fix the demo call-sites to pass extensions to the compiler (once the pipeline lands), and add one end-to-end test that asserts an extension table appears in a compiled MSI.

P1 — onboarding / docs (the actual ask)

Add a "Concepts" section to the manual right after Overview: what a Component is (and the component rule), ProductCode vs UpgradeCode vs PackageCode, why elevation exists, sequencing basics. This is the missing piece for a WiX-newcomer.
Document forge init + templates in the manual (§14) — cheap, fixes the README/doc divergence.
Link docs/tutorials/ from the README — unbury the content that already teaches.
Emit a shortcut in the forge init scaffold — a 3-line package.Shortcut(...); it's the most common first ask.
Write one end-to-end "AcmeSuite" enterprise walkthrough that composes the capabilities (after P0 makes IIS/SQL real).
Expose feature-gating for service/registry via the fluent API, or document the current limitation explicitly.

P2 — polish (mostly gated on going public)

Publish alpha packages to nuget.org (or reword the README's Get-Started to the real clone+build path) so day-1 commands aren't dead. Gated on the private→public flip.
Add a "new here? start with these 4 demos" callout above the feature matrices; sync stale doc counts; fix the non-compiling §22 .Product(...) snippet; warn on define-without-schedule custom actions.

7 · FalkForge vs WiX for a total beginner

WiX drops a newcomer straight into XML authoring and the Windows Installer spec — Components/Features/Directories tables, GUID discipline, ICE validation, WiX's own quirky syntax — with sparse official docs and a notoriously steep, tribal-knowledge learning curve (most people learn it by scraping Stack Overflow). FalkForge's fluent C# API removes the XML pain and gives type-safety and IntelliSense, and its README + demo catalog are more approachable on day one. Net: FalkForge is meaningfully easier to start (fewer footguns, better first 15 minutes). But under the hood it's still Windows Installer, and the manual currently assumes the same mental model WiX assumes — just without teaching it. Add the Concepts section (P1) and fix the extension pipeline (P0), and the advantage over WiX widens sharply; without them, the honest framing is "WiX's hard parts wearing a friendlier syntax — plus, right now, extensions that don't emit."

8 · Method & evidence

Four independent read-only scouts walked the actual newcomer entry points (README → install → forge init/templates → manual → demos) for the simple and enterprise journeys, plus a documentation-meatiness pass. A fifth agent verified the extension claim by static code trace, and a sixth built and inspected real MSIs on this Windows box to convert the claim from code-review to live proof (table lists via forge inspect + a WindowsInstaller COM row query, with demo 01 as a positive control). Every finding above carries a file:line or a build+inspect result. Nothing here is speculative; where scope is uncertain (e.g. extensions beyond SQL/IIS that share the contributor path) it is called out as "by the same code path", not claimed as separately tested.
