## Workstream A — Performance & allocation remediation

From the 2026-07-07 sweep. Each item follows normal TDD/gates; HIGH items first. Line numbers are as of commit `00ecf4e`.

### HIGH

- **A1. ✅ DONE 2026-07-07 (merge `a09e9bf`).** Bundle payload double-decompression on every install.
  `src/FalkForge.Engine.Protocol/Bundle/BundleReader.cs:118-137` (`Extract`) + `221-246` (`ExtractPayload`); consumers `src/FalkForge.Engine/Program.cs:107,152,320,360`, `src/FalkForge.Cli/Commands/ExtractCommand.cs:76,116`, `PlanCommand.cs:65`, `PlanDiffCommand.cs:81,88`, `VerifyCommand.cs:243`.
  `Extract()` decompresses and SHA-256-verifies every payload fully in memory, discards the result, then `ExtractPayload()` decompresses each payload again and writes via `File.WriteAllBytesAsync`. Every install pays 2× full decompression and ~3× payload-size transient memory. `--extract-list` decompresses the whole bundle just to print names/sizes already present in the TOC.
  Fix: split into "read TOC only" (no payload decode) vs a single-pass verify+extract that streams decompressed bytes to the output file while hashing incrementally (`IncrementalHash`/`CryptoStream`); list mode must never touch payload bytes.

- **A2. Bundle compiler holds ALL compressed payloads in memory before writing.**
  `src/FalkForge.Compiler.Bundle/Compilation/PayloadEmbedder.cs:43,63-79` and `DeltaBundleCompiler.cs:225,270-292`.
  `compressedData = new byte[count][]` filled by `Parallel.For`, then written sequentially — peak memory is the sum of all compressed payload sizes. Fix: stream each payload to the output (or per-payload temp file) as its compression completes instead of buffering everything.

- **A3. Delta compilation loads whole files into memory, N-way parallel.**
  `DeltaBundleCompiler.cs:47-52` (caches every old payload's full decompressed bytes, even for packages that need no delta) and `:123` (`File.ReadAllBytes` inside `Parallel.ForEach`). Root cause: `Compression/DeltaCompressor.cs:14,46` and `Engine/Download/DeltaApplicator.cs:12` take `byte[]` and wrap in `MemoryStream`, while Octodiff accepts `Stream`. Fix: change signatures to `Stream`/paths and pass `FileStream` straight through.
  Note: `DeltaApplicator` in Engine/Download appears to have no callers in src — confirm; if dead, remove instead of optimizing.

- **A4. `MsiProgressState.ParseFields` allocates a `Dictionary<int,int>` on every MSI progress callback tick.**
  `src/FalkForge.Engine/Execution/MsiProgressState.cs:65-93` via `ProcessMessage` (line 19). Fires dozens-to-thousands of times per install. Fields are bounded (indices 1–3). Fix: fixed-slot struct or inline array, zero alloc.

- **A5. `MessageSerializer.Serialize` allocates `MemoryStream` + `BinaryWriter` + `ToArray()` copy per pipe message.**
  `src/FalkForge.Engine.Protocol/Serialization/MessageSerializer.cs:44-73`. Progress/log/phase messages fire constantly during install. Fix: pooled/reusable buffer (`ArrayBufferWriter<byte>` or equivalent).

- **A6. `MessageDeserializer.Deserialize` double-copies the receive buffer.**
  `src/FalkForge.Engine.Protocol/Serialization/MessageDeserializer.cs:33-44` — `bytes.ToArray()` copies a buffer the caller already rented from `ArrayPool` (`Transport/PipeTransportBase.cs:83-89`). Fix: accept `ReadOnlyMemory<byte>` and wrap the rented array without copying.

- **A7. `DirectoryTreeSynthesizer.ComputeDirectoryId` re-hashes the whole ancestor chain per call, no cache.**
  `src/FalkForge.Compiler.Msi/Recipe/DirectoryTreeSynthesizer.cs:29-65`; callers `ComponentTableProducer.cs:77` (per component) and `DirectoryTableProducer.cs:174` (per segment per component/file). SHA-256 + string concat redone per file for shared directories. Fix: memoize on `RecipeBuildContext` keyed by `(root token, relative path)` — same pattern as `GetOrBuildFileToComponentMap` (`RecipeBuildContext.cs:21-24`).

- **A8. `ComponentResolver` sanitizes the same filename twice per file.**
  `src/FalkForge.Compiler.Msi/ComponentResolver.cs:117,123` (called from `:42/:54` and `:76/:88`). Fix: sanitize once, pass to both ID generators.

### MED

- **A9. `SanitizeId` always allocates even when input is already clean.**
  `ComponentResolver.cs:136-146` and `DirectoryTreeSynthesizer.cs:114-124`. Apply the pre-scan pattern already used in `Recipe/Producers/ProducerHelpers.cs:73-97` (`SanitizeDirectoryId`).

- **A10. `InstallPath` re-splits and re-formats on every access.**
  `src/FalkForge.Core/InstallPath.cs:17` (`Segments` splits fresh per get) and `:24-27` (`ToString()` allocates per call). Compounds A7. Fix: lazy-cache both on the immutable instance.

- **A11. `MsiAuthoring` reads the whole compiled MSI into memory just to hash it (WinGet path).**
  `src/FalkForge.Compiler.Msi/Recipe/MsiAuthoring.cs:351` — `SHA256.HashData(File.ReadAllBytes(...))`. Fix: stream via `File.OpenRead`.

- **A12. `PipeTransportBase.SendAsync` allocates a 4-byte length array per send.**
  `src/FalkForge.Engine.Protocol/Transport/PipeTransportBase.cs:55` (`BitConverter.GetBytes`). Fix: combine length prefix + payload into one rented buffer and issue a single `WriteAsync` (also halves syscalls). Pairs naturally with A5.

### LOW (batch opportunistically alongside neighbouring work)

- **A13.** `GuidUtility.CreateDeterministicGuid` allocates 4 small arrays per call (`src/FalkForge.Core/GuidUtility.cs:19-38`); stackalloc + `Guid.TryWriteBytes` + `SHA256.HashData(span, span)`. Meaningful only at thousands-of-files scale; do together with A7/A8.
- **A14.** `ParallelCabinetBuilder.cs:57` LINQ `(item, index)` tuple staging — replace with plain indexed loop.
- **A15.** `IntegritySbomGenerator.cs:75,128` `Encoding.UTF8.GetString(ms.ToArray())` double-copy — use `TryGetBuffer`; match `CycloneDxSbomGenerator`'s stream-direct approach.
- **A16.** `CSharpEmitter.cs:388` and `BundleCSharpEmitter.cs:34` `new string(' ', n)` per emitted line — use `StringBuilder.Append(char, count)`.
- **A17.** `DeltaApplicator.Apply` (`src/FalkForge.Engine/Download/DeltaApplicator.cs:26`) `ToArray()` of full reconstructed payload — moot if A3's dead-code check removes the type.

### Verified clean (do not churn)

`ConditionLexer`/`ConditionEvaluator`, codec registry (`FrozenDictionary`), `EngineLogger`, `PayloadDownloader`, `CacheLayout`/`PackageCache` hashing, `BundleDetacher` (model implementation: ArrayPool + stackalloc + streaming), `CabinetBuilder`, `FileTableProducer`, `UiStringResolver`, WPF converters, Extensions table contributors. `RollbackJournal`'s per-entry flush is an intentional durability tradeoff — leave it.
