### HIGH

- **A1. ✅ DONE 2026-07-07 (merge `a09e9bf`).** Bundle payload double-decompression on every install.
  `src/FalkForge.Engine.Protocol/Bundle/BundleReader.cs:118-137` (`Extract`) + `221-246` (`ExtractPayload`); consumers `src/FalkForge.Engine/Program.cs:107,152,320,360`, `src/FalkForge.Cli/Commands/ExtractCommand.cs:76,116`, `PlanCommand.cs:65`, `PlanDiffCommand.cs:81,88`, `VerifyCommand.cs:243`.
  `Extract()` decompresses and SHA-256-verifies every payload fully in memory, discards the result, then `ExtractPayload()` decompresses each payload again and writes via `File.WriteAllBytesAsync`. Every install pays 2× full decompression and ~3× payload-size transient memory. `--extract-list` decompresses the whole bundle just to print names/sizes already present in the TOC.
  Fix: split into "read TOC only" (no payload decode) vs a single-pass verify+extract that streams decompressed bytes to the output file while hashing incrementally (`IncrementalHash`/`CryptoStream`); list mode must never touch payload bytes.

- **A2. ✅ DONE 2026-07-08 (merge `1a83cad`).** Bundle compiler held ALL compressed payloads in memory before writing (`PayloadEmbedder`). Now each payload compresses to a per-payload temp file (parallelism kept) and streams to the output through one pooled 64 KiB buffer; temp dir cleaned in finally. Peak memory no longer scales with payload count/size. Byte-identical output (order/gzip level/offsets/TOC verified). +3 tests.

- **A3. ✅ DONE 2026-07-08 (merge `6a70d92`).** Delta compilation loaded whole files into memory N-way parallel. Now `DeltaCompressor` takes `Stream`s (FileStream straight to Octodiff), old payloads extracted to temp lazily only when a delta is actually produced, compressed payloads streamed via per-payload temp files. Byte-identical delta bytes + base/reconstructed SHA-256 + threshold. +4 tests (incl. temp-dir cleanup on success/failure).
  Also closes **A17** — `Engine/Download/DeltaApplicator` confirmed dead (no non-test callers) and removed.

- **A4. ✅ DONE (merge `24476d3`).** `MsiProgressState.ParseFields` allocates a `Dictionary<int,int>` on every MSI progress callback tick.
  `src/FalkForge.Engine/Execution/MsiProgressState.cs:65-93` via `ProcessMessage` (line 19). Fires dozens-to-thousands of times per install. Fields are bounded (indices 1–3). Fix: fixed-slot struct or inline array, zero alloc.

- **A5. `MessageSerializer.Serialize` allocates `MemoryStream` + `BinaryWriter` + `ToArray()` copy per pipe message.**
  `src/FalkForge.Engine.Protocol/Serialization/MessageSerializer.cs:44-73`. Progress/log/phase messages fire constantly during install. Fix: pooled/reusable buffer (`ArrayBufferWriter<byte>` or equivalent).

- **A6. `MessageDeserializer.Deserialize` double-copies the receive buffer.**
  `src/FalkForge.Engine.Protocol/Serialization/MessageDeserializer.cs:33-44` — `bytes.ToArray()` copies a buffer the caller already rented from `ArrayPool` (`Transport/PipeTransportBase.cs:83-89`). Fix: accept `ReadOnlyMemory<byte>` and wrap the rented array without copying.

- **A7. `DirectoryTreeSynthesizer.ComputeDirectoryId` re-hashes the whole ancestor chain per call, no cache.**
  `src/FalkForge.Compiler.Msi/Recipe/DirectoryTreeSynthesizer.cs:29-65`; callers `ComponentTableProducer.cs:77` (per component) and `DirectoryTableProducer.cs:174` (per segment per component/file). SHA-256 + string concat redone per file for shared directories. Fix: memoize on `RecipeBuildContext` keyed by `(root token, relative path)` — same pattern as `GetOrBuildFileToComponentMap` (`RecipeBuildContext.cs:21-24`).

- **A8. `ComponentResolver` sanitizes the same filename twice per file.**
  `src/FalkForge.Compiler.Msi/ComponentResolver.cs:117,123` (called from `:42/:54` and `:76/:88`). Fix: sanitize once, pass to both ID generators.
