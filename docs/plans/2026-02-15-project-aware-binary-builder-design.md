# Project-Aware Binary Builder Design

**Date:** 2026-02-15
**Status:** Proposed

## Problem

FalkForge users must hardcode absolute paths when referencing binaries and installer outputs across projects:

```csharp
// Custom action DLL -- brittle, CI-unfriendly
p.Binary("CA", @"C:\build\output\MyCustomAction.dll");

// Bundle referencing MSI -- breaks on every machine
b.MsiPackage(@"C:\build\output\App.Installer.msi", pkg => { ... });
```

These hardcoded paths break across developer machines, CI agents, and configuration changes. When the referenced projects live in the same solution, the build system already knows their output paths -- there is no reason to force users to specify them manually.

## Solution

An MSBuild source-generation pipeline in the FalkForge SDK that:

1. Discovers all `<ProjectReference>` items that are FalkForge projects
2. Queries each for its output type and artifact path
3. Generates a `ProjectOutputs.g.cs` file with `const string` properties for each discovered output
4. Compiles the generated file into the referencing project automatically

This gives users strongly-typed, always-correct paths:

```csharp
// Resolved automatically from ProjectReference metadata
p.CustomAction(ProjectOutputs.MyCustomAction, nameof(MyCustomActions.DoStuff), ca =>
{
    ca.Deferred().NoImpersonate();
    ca.After = "InstallFiles";
});

b.MsiPackage(ProjectOutputs.AppInstaller, pkg => { ... });
```

## Architecture

### MSBuild Target Flow

```
┌──────────────────────────────────────────────────────┐
│  Suite.Bundle.csproj (FalkOutputType=Bundle)         │
│                                                       │
│  <ProjectReference Include="..\App.Installer\...">   │
│  <ProjectReference Include="..\Svc.Installer\...">   │
└──────────────────────┬───────────────────────────────┘
                       │ BeforeTargets="CoreCompile"
                       ▼
              _GenerateProjectOutputs
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
   _GetFalkOutput  _GetFalkOutput  _GetFalkOutput
   (App.Installer) (Svc.Installer) (MyCustomAction)
          │            │            │
          ▼            ▼            ▼
   _ComputeFalkArtifactPath  (runs in each referenced project)
          │
          ▼
   _WriteFalkProjectOutputsSource  (inline Roslyn task)
          │
          ▼
   $(IntermediateOutputPath)ProjectOutputs.g.cs
          │
          ▼
   Added to <Compile> items → compiled into assembly
```

### Target Responsibilities

| Target | Scope | Purpose |
|--------|-------|---------|
| `_ComputeFalkArtifactPath` | Per-project | Computes artifact path from `FalkOutputType` |
| `_GetFalkForgeOutput` | Per-project | Exports `_FalkOutput` item with metadata |
| `_GenerateProjectOutputs` | Referencing project | Orchestrates cross-project queries, invokes code gen |
| `_WriteFalkProjectOutputsSource` | Inline task | Writes the `.g.cs` file to `obj/` |

### Output Path Convention

Each `FalkOutputType` maps to a deterministic artifact path:

| FalkOutputType | Artifact Path | Extension |
|----------------|---------------|-----------|
| `Msi` | `$(TargetDir)$(AssemblyName).msi` | `.msi` |
| `Bundle` | `$(TargetDir)$(AssemblyName).exe` | `.exe` |
| `Module` | `$(TargetDir)$(AssemblyName).msm` | `.msm` |
| `Patch` | `$(TargetDir)$(AssemblyName).msp` | `.msp` |
| `CustomAction` | `$(TargetPath)` | `.dll` |

These conventions match what the `FalkGenerateInstaller` target already produces.

## Generated Code

The `_GenerateProjectOutputs` target produces a file at `$(IntermediateOutputPath)ProjectOutputs.g.cs`:

```csharp
// <auto-generated/>
// Generated by FalkForge SDK. Do not edit manually.
namespace FalkForge;

internal static class ProjectOutputs
{
    /// <summary>Msi output of App.Installer</summary>
    internal const string AppInstaller = @"D:\repos\MySolution\App.Installer\bin\Release\net10.0-windows\App.Installer.msi";

    /// <summary>Msi output of Service.Installer</summary>
    internal const string ServiceInstaller = @"D:\repos\MySolution\Service.Installer\bin\Release\net10.0-windows\Service.Installer.msi";

    /// <summary>CustomAction output of MyCustomAction</summary>
    internal const string MyCustomAction = @"D:\repos\MySolution\MyCustomAction\bin\Release\net10.0-windows\MyCustomAction.dll";
}
```

### Name Sanitization

Project names are sanitized to valid C# identifiers:

| Project Name | Property Name |
|--------------|---------------|
| `App.Installer` | `App_Installer` |
| `My-CustomAction` | `My_CustomAction` |
| `Service Worker` | `Service_Worker` |
| `123Project` | `_123Project` (leading digit gets underscore prefix) |

Characters `.`, `-`, and ` ` are replaced with `_`. A leading digit gets a `_` prefix.

## API Changes

### New CustomAction Overload on PackageBuilder

```csharp
/// <summary>
/// Registers a binary and creates a DLL custom action in one call.
/// The binary name is derived from the filename of <paramref name="binaryPath"/>.
/// </summary>
/// <param name="binaryPath">
/// Path to the custom action DLL. Use <c>ProjectOutputs.MyCustomAction</c>
/// for solution-local references.
/// </param>
/// <param name="entryPoint">
/// The exported function name (e.g., <c>nameof(MyCustomActions.DoStuff)</c>).
/// </param>
/// <param name="configure">Optional builder configuration.</param>
public PackageBuilder CustomAction(
    string binaryPath,
    string entryPoint,
    Action<CustomActionBuilder>? configure = null)
{
    var binaryName = Path.GetFileNameWithoutExtension(binaryPath);
    Binary(binaryName, binaryPath);

    return CustomAction($"{binaryName}.{entryPoint}", ca =>
    {
        ca.Type = CustomActionType.DllFromBinary;
        ca.BinaryKey = binaryName;
        ca.DllEntryPoint = entryPoint;
        configure?.Invoke(ca);
    });
}
```

### Usage Examples

**Custom action (before):**
```csharp
p.Binary("CA", @"C:\build\MyCustomAction.dll");
p.CustomAction("CA.DoStuff", ca =>
{
    ca.Type = CustomActionType.DllFromBinary;
    ca.BinaryKey = "CA";
    ca.DllEntryPoint = "DoStuff";
    ca.Deferred().NoImpersonate();
    ca.After = "InstallFiles";
});
```

**Custom action (after):**
```csharp
p.CustomAction(ProjectOutputs.MyCustomAction, nameof(MyCustomActions.DoStuff), ca =>
{
    ca.Deferred().NoImpersonate();
    ca.After = "InstallFiles";
});
```

**Bundle MSI reference (before):**
```csharp
b.MsiPackage(@"C:\build\App.Installer.msi", pkg =>
{
    pkg.DisplayName = "Application";
});
```

**Bundle MSI reference (after):**
```csharp
b.MsiPackage(ProjectOutputs.AppInstaller, pkg =>
{
    pkg.DisplayName = "Application";
});
```

## Build Workflow

### Single-MSI Project

```
MySolution/
  App.Installer/          FalkOutputType=Msi
  App.Installer.csproj    <Sdk Name="FalkForge.Sdk" />
```

```bash
dotnet build              # Compiles + generates App.Installer.msi
dotnet run --project App.Installer  # Same, via explicit project
```

### Bundle with Multiple MSI Projects

```
MySolution/
  App.Installer/          FalkOutputType=Msi
  Service.Installer/      FalkOutputType=Msi
  MyCustomAction/         FalkOutputType=CustomAction
  Suite.Bundle/           FalkOutputType=Bundle
    ProjectReferences -> App.Installer, Service.Installer, MyCustomAction
```

```bash
dotnet build
```

MSBuild resolves the build order from `ProjectReference`:

1. `MyCustomAction` builds (produces DLL)
2. `App.Installer` builds (produces MSI)
3. `Service.Installer` builds (produces MSI)
4. `Suite.Bundle` builds:
   - `_GenerateProjectOutputs` queries all three references
   - Generates `ProjectOutputs.g.cs` with paths to all artifacts
   - Compiles (the C# code uses `ProjectOutputs.AppInstaller`, etc.)
   - `FalkGenerateInstaller` produces the bundle EXE

### CI Pipeline

No special CI configuration needed. The paths are resolved from MSBuild properties that respect `$(Configuration)`, `$(RuntimeIdentifier)`, and `$(OutputPath)`:

```yaml
- dotnet build -c Release
- dotnet test
# Artifacts are at predictable locations via ProjectOutputs
```

## Demo Structure

A reference solution demonstrating the feature:

```
samples/
  ProjectAwareDemo/
    ProjectAwareDemo.sln
    src/
      App/                          # The application being installed
        App.csproj
        Program.cs

      App.CustomActions/            # Custom action DLL
        App.CustomActions.csproj    # FalkOutputType=CustomAction
        CustomActions.cs            # Exported functions

      App.Installer/                # MSI installer project
        App.Installer.csproj        # FalkOutputType=Msi
                                    # ProjectReference -> App, App.CustomActions
        InstallerDefinition.cs      # Uses ProjectOutputs.App_CustomActions

      Service.Installer/            # Second MSI
        Service.Installer.csproj    # FalkOutputType=Msi
        InstallerDefinition.cs

      Suite.Bundle/                 # Bundle combining both MSIs
        Suite.Bundle.csproj         # FalkOutputType=Bundle
                                    # ProjectReference -> App.Installer, Service.Installer
        BundleDefinition.cs         # Uses ProjectOutputs.AppInstaller, etc.
```

**App.Installer/InstallerDefinition.cs:**
```csharp
using FalkForge;
using FalkForge.Builders;

Installer.Build(p =>
{
    p.Name = "My Application";
    p.Version = new Version(1, 0, 0);
    p.Manufacturer = "Contoso";

    p.Feature("Main", f =>
    {
        f.FileSet("AppFiles", fs =>
        {
            fs.SourceDirectory = @"..\App\bin\Release\net10.0-windows\publish";
            fs.TargetDirectory = "INSTALLFOLDER";
        });
    });

    // One-liner custom action using project-aware path
    p.CustomAction(ProjectOutputs.App_CustomActions, "ConfigureApp", ca =>
    {
        ca.Deferred().NoImpersonate();
        ca.After = "InstallFiles";
    });
});
```

**Suite.Bundle/BundleDefinition.cs:**
```csharp
using FalkForge;
using FalkForge.Builders;

Installer.BuildBundle(b =>
{
    b.Name = "Contoso Suite";
    b.Version = new Version(1, 0, 0);
    b.Manufacturer = "Contoso";

    b.Chain(chain =>
    {
        chain.MsiPackage(ProjectOutputs.AppInstaller, pkg =>
        {
            pkg.DisplayName = "Application";
        });

        chain.MsiPackage(ProjectOutputs.ServiceInstaller, pkg =>
        {
            pkg.DisplayName = "Background Service";
        });
    });
});
```

## Design Decisions

### Why const strings instead of a typed wrapper?

`const string` values are embedded at compile time, have zero runtime overhead, and work everywhere a `string` is accepted. A wrapper type would add API surface and require callers to unwrap.

### Why generate at build time instead of using analyzers?

MSBuild targets can query cross-project metadata via the `<MSBuild>` task. Roslyn analyzers run in the compiler process and cannot access MSBuild project metadata from other projects. The target approach is the standard pattern used by NuGet, the .NET SDK, and WiX.

### Why namespace FalkForge instead of project-specific?

A single well-known namespace means users do not need a `using` directive in most cases (the FalkForge namespace is already imported). The class is `internal` so there is no conflict risk across assemblies.

### Why RoslynCodeTaskFactory?

It is available in all modern .NET SDK versions, supports inline C# code in MSBuild, and avoids shipping a separate task assembly. The generated code is trivial string concatenation.
